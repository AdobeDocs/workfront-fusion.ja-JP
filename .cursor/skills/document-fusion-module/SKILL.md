# Fusion コネクタモジュールのドキュメント化

このスキルを使用して、1つ以上の新しいモジュールをWorkfront Fusion コネクタ記事に追加します（例：`adobe-firefly-modules.md`、`adobe-photoshop-modules.md`、`azure-dev-ops.md`）。

## 必要な入力（モジュールごと）

モジュールのフィールドを入力する前に、ユーザー&#x200B;**は次のすべてを提供する必要があります。**&#x200B;欠けている部分があれば、先に進む前に停止して要求します。

1. **モジュール名** — Fusion UIに表示されている通り（例：`Generate adaptive composite`）。
2. **コネクタ記事のパス** — エージェントがこれを見つけ、ユーザーと確認します（フェーズ 1を参照）。
3. **基になるAPI操作のAPI URL**。 これは、各フィールドの目的をAPI仕様に対して相互参照できるようにするために必要です。
4. 標準ビュー&#x200B;**（`Show advanced settings`** オフ **）のモジュール設定パネルの** スクリーンショット。
5. **高度なビュー** （`Show advanced settings` **checked**）のモジュールの設定パネルのスクリーンショット、またはモジュールに高度なフィールドがないことを示す明示的なステートメント。

## ワークフロー

これは多段階のプロセスです。 ペーシングは重要です。ユーザーの前ではなく、*で*&#x200B;作業してください。 各フェーズの境界で進捗を確認し、いつでも修正に対してオープンな状態を維持します。

### フェーズ 1 – 作業範囲の決定

1. **これが1つのモジュールか複数のモジュールかを確認します**: *「1つのモジュールまたは複数のモジュールをコネクタ記事に追加していますか？」*

2. **回答に基づいて、すべてのモジュール名を事前に**&#x200B;収集します。
   - **単一モジュール**:Fusion UIに表示される名前と同じ名前を求めます。
   - **複数のモジュール**：すべての名前を一度に尋ねるか、それらを一覧表示するコネクタのスクリーンショットを求めます（たとえば、各モジュールのタイルラベルを表示するFusionのコネクタ概要画面）。 いずれにしても、追加する新しいモジュールのリストを確認して、この手順を終了します。

3. **モジュールで示されるコネクタ名に基づいて`help/workfront-fusion/references/apps-and-modules/`を検索して、自分でコネクタ記事を探します**。 `Glob`または`Grep` ツールを使用します。

4. **ユーザー**&#x200B;で記事のパスを確認します：*&quot;モジュール名に基づいて、これは`help/workfront-fusion/references/apps-and-modules/adobe-connectors/adobe-firefly-modules.md`になります。 正解ですか？&quot;*

5. **コネクタの記事**&#x200B;を読んで、既存の構造と規則を確認してください。 次の点に注意してください。
   - 見出しのスタイル （文頭の場合は通常`### Module name`）
   - 既存のモジュールの順序（通常はアルファベット順）
   - `Connection`行の文言
   - ネストされたフィールドの書き込み方法（例：`Image > Source`）
   - 既存の脚注または注釈の規則

### フェーズ 2：各モジュールのプレースホルダーを前面に配置する

新しいモジュールが1つしかない場合でも、明確な視覚的なロードマップを構築できます。 新しいモジュールが複数ある場合は、それらをすべて配置します。

新しいモジュールごとに、アルファベット順にプレースホルダーセクションを挿入します。

```markdown
### Module name

<!-- PLACEHOLDER: Add description of the Module name module here. -->

<table style="table-layout:auto"> 
 <col> 
 <col> 
 <tbody> 
  <tr> 
   <td role="rowheader">[!UICONTROL Connection]</td> 
   <td>For instructions on creating a connection to [!DNL Connector Name], see <a href="#create-a-connection-to-connector-name" class="MCXref xref" >Create a connection to [!DNL Connector Name]</a> in this article.</td> 
  </tr> 
  <!-- PLACEHOLDER: Add field rows for the Module name module here. -->
 </tbody> 
</table>
```

追加されたプレースホルダーのリストをユーザーに表示します（例：「アダプティブコンポジットの生成」、「Image5を使用した画像の生成」、「正確なコンポジットの生成」、「ビデオの生成」）。 次に、どの語句で始めるかを尋ねます：*「どの語句で始めますか？」*

### フェーズ 3 - モジュールごとのドキュメントループ

各モジュールについて、次のモジュールに進む前に、このループを完全に完了してください。

#### 3a. API URLの取得

ユーザーにAPI URLを尋ねます：*「`<module name>`の基になるAPI操作のAPI URLを共有してください。」*

お持ちの場合：
- URLで`WebFetch`をお試しください。
- ページが空/JS レンダリングされている場合（Adobe Developer ドキュメントに共通）、関連する機能ガイドページ（通常は`.../guides/how-tos/...` URL）を探し、代わりに取得します。
- それでも問題が解決しない場合は、操作名として`WebSearch`にフォールバックします。

後でフィールドの説明を行う際は、バックグラウンドコンテキストとして学習した内容を使用します。 ユーザーにダンプしないでください。簡単に確認してください：*「API コンテキストを取得しました – 標準ビューのスクリーンショットの準備ができました」*

#### 3b. 標準ビューのスクリーンショットを取得

質問：*「モジュールの設定パネルのスクリーンショットを`Show advanced settings`**オフ**と共有してください。 標準フィールドを処理した後、高度なビューを要求しますが、まだ送信しません。&quot;*

#### 3c: 標準フィールドのドキュメント化

標準ビューのスクリーンショットから作業して、表示されているすべてのフィールドを上から下にキャプチャします。 各フィールドについて：

1. フィールドの正確なUI ラベルを行ヘッダーとして使用します。 グループ化されたフィールドの場合は、` > `で結合します。

   ```
   [!UICONTROL Background > Image > Source]
   ```
2. スクリーンショットのフィールド（グレー/イエローのキャプション）の下にあるヘルパーテキストを読みます。 説明の基礎として使用してください。
3. 手順3aのAPI コンテキストを相互参照して、フィールドが何を表しているかを特定します（例えば、`background.fillAreaMask`は「オブジェクトが配置される背景の領域」です）。
4. **フィールドが何か** （APIから）と&#x200B;**それを提供する方法** （UI ヘルパーテキストとフィールドタイプから）を組み合わせて、説明を記述します。

APIがスクリーンショットをサポートしている場合でも、スクリーンショットに表示されないフィールドはスキップします。 フィールドを作成しない。

標準フィールドを配置した後、簡単にユーザーに要約します（例：「標準フィールドが追加されました：接続、プロンプト、縦横比、...」）。その後、手順3dに移動します。

#### 3d: アドバンスドビューのスクリーンショットを取得

質問：*「モジュールの設定パネルのスクリーンショットを`Show advanced settings`**checked**と共有してください。 モジュールに高度なフィールドがない場合は、そのフィールドを省略します。&quot;*

#### 3e. 詳細フィールドのドキュメント化

詳細表示のスクリーンショットから作業：

1. 標準ビューに既に&#x200B;*が*&#x200B;にないフィールドを特定します。これらは詳細フィールドです。
2. 詳細フィールドごとに、手順3cと同じ説明プロセスに従います。
3. `<td role="rowheader">`内のフィールド名に`*`を追加します：

   ```html
   <td role="rowheader">[!UICONTROL Seeds]*</td>
   ```
4. `</table>`を閉じた後、次の脚注を独自の行に追加します。

   ```markdown
   * These fields are advanced fields, and do not display unless you select **[!UICONTROL Show advanced settings]**.
   ```

ユーザーがモジュールに高度なフィールドがないと言った場合は、手順3dと3eを完全にスキップします。

#### 3f. モジュールの説明のドラフト

ここで（以前ではない）、セクションの上部にある説明プレースホルダーを、API コンテキストとフィールドが意味するものに基づく1文のドラフトに置き換えます。 ユーザーのレビュー用のドラフトとして常に提供します。

> *&quot;この説明を作成しました：&#39;...&#39; それを使用するか、編集するか、置き換えるか？&quot;*

#### 3g回線： モジュールごとのラップアップ

モジュールのドキュメント化されたフィールド（標準+詳細）の最終的な概要をユーザーに提供し、開いている質問を表示します。

- スクリーンショットでフィールドが切り取られましたか？
- モジュールの説明は受け入れられますか？
- 非推奨（廃止予定）または注目に値するフラグを設定する必要がありますか？

モジュールが完了したことを伝える前に、確認を待ちます。 次の質問：*「次のモジュールに進む準備ができましたか？」* （または「このコネクタの記事はすべて完了です」）。

### フェーズ 4 – 最終検証

すべてのモジュールを文書化したら、次のチェックリストを各モジュールに対して実行します。

- [ ] モジュールの見出しは`### ` （H3）で、文頭の大文字と小文字が使用されます。
- [ ] モジュールは、兄弟モジュールに対してアルファベット順に表示されます。
- [ ]最初の行は`Connection`で、標準の接続リンクの文言を使用します。
- [ ]すべてのドキュメント済みフィールドは、ユーザーのスクリーンショットに表示されます。
- [ ]詳細フィールドには`*`とマークされ、テーブルの下に1つの脚注が存在します。
- [ ] APIだけで作成されたフィールドはありません。
- [ ] Source ドロップダウンの説明では、正確なUI オプションラベルを使用します（以下の「Source フィールドの規則」を参照）。
- [ ] モジュールの説明は、モジュールの機能を説明する1文の要約です。

## Sourceのフィールド規則

Fusion コネクタの画像ソースフィールドには、通常、2つのオプションが表示されます。 **ユーザーのスクリーンショットに表示されている正確なラベルを使用する** – 様々な世代のモジュールで様々な文言が使用されます。

| ドロップダウンが表示される場合… | この形式を使用 |
|---|---|
| **画像をアップロード**&#x200B;および&#x200B;**画像URL** （新しいモジュール） | `<ul><li><b>Upload Image</b><p>Upload the image, or map the image file from a previous module.</p></li><li><b>Image URL</b><p>Enter or map the URL of the image.</p></li></ul>` |
| **ファイル**&#x200B;および&#x200B;**署名済みURL** （古いモジュール） | `<ul><li><b>File</b><p>Select a source file from a previous module, or map the source file's reference image file name and reference image file.</p></li><li><b>Presigned URL</b><p>Enter or map the URL of the source image.</p></li></ul>` |

スクリーンショットにドロップダウンオプションが表示されない場合は、ユーザーに共有を依頼します。

## フィールドの説明スタイル

- 同じ記事の既存のモジュールエントリの音声と構造を一致させます。
- 説明は簡潔にする：通常は、1つのフィールドにつき1つか2つの文章で十分です。
- 別のフィールドから参照されている任意のUI フィールド名には、`[!UICONTROL ...]` マークアップを使用します。
- リテラル値（`<code>en-US</code>`、`<code>0.0</code>`）には`<code>...</code>`を使用します。
- 数値範囲フィールドの場合は、許可される範囲を明示的に記述します。*「0 ～ 1...」の数値を入力します。*
- 個別のオプションを含むドロップダウンの場合は、`<ul>`として列挙し、`<b>Option</b>`とその後に`<p>`の説明を付けます。

## 共通フィールドと推奨される説明

モジュール間の一貫性を保つために、次の方法を再利用します。

**Seeds** （フィールドで項目の追加を介して複数のシード値が許可されている場合）:
> 「**項目を追加**」をクリックしてシード値を追加し、整数を入力またはマッピングします。 バリエーションごとに1つのシードを使用します。 シード値の数は、両方が指定されている場合、**バリエーション数**&#x200B;の値と一致する必要があります。

**バリエーションの数** （スクリーンショットの実際の範囲に範囲を置き換えます）:
> [min]から[max]までの数字を入力してください。 モジュールは、この数の[出力タイプ ]のバリエーションを生成します。

**制限** （標準のFusion実行サイクル制限。スクリーンショットに表示される場合のみ含める）:
> 1回の実行サイクル中にモジュールが処理する結果の最大数を入力またはマッピングします。

**ロケール**:
> 言語と地域コードを入力またはマッピングして、生成されたコンテンツを特定の国と言語に合わせてカスタマイズします。 ロケールは、ISO 639-1言語コードおよびISO 3166-1地域で指定する必要があります。 例：`en-US`

## いつでも修正に対してオープン

ユーザーは、フローの途中で作業を修正することもできます。シードの文言、ドロップダウンラベル、フィールドの配置、高度な分割や標準分割など。修正を承認済みとして扱い、プッシュバックすることなく更新します。 修正後、変更された内容を簡単にまとめます。

ユーザーが新しいコンベンションのミッドフローを導入した場合（例：「高度なフィールドに`*`を付けて脚注を追加しましょう」）、セッションの残りの部分にそれを採用し、既に文書化されているモジュールに過去にさかのぼって適用します。

## 情報が競合している場合の対処方法

API仕様とUIが一致しない場合（これは頻繁に発生します）:

1. **UI**&#x200B;を信頼します。これは、ユーザーがFusionで実際に目にするものです。
2. **ユーザーへの返信の不一致**&#x200B;をメモして、さらに調査するかどうかをユーザーが決定できるようにします。
3. **記事の本文に説明**&#x200B;を作成しないでください。UIの内容を事実に基づいて説明してください。

ユーザーからの2つのスクリーンショットが互いに矛盾している場合（例えば、1つは`Blend`を示し、別のものは同じモジュールの`Harmonization`を示す）、停止し、書き込む前にどれが正しいかを尋ねます。 API仕様を相互参照することで、異なるモジュールに属する可能性が高いスクリーンショットを特定します。
